# Story 2.6: Downtime -- Outcome Approval and Application

**Status:** Draft

**Priority: HIGH -- soft deadline 8 April 2026**

## Story

**As a** Storyteller,
**I want** to approve, modify, or reject individual downtime outcomes and have approved outcomes write back to character data automatically,
**so that** a full downtime cycle resolves without hand-editing JSON.

## Context: What the Downtime Helper Already Has

- ✅ Territory influence table (Territories tab): spending per character per territory is already displayed
- ✅ Ambience score calculation: roll successes from sphere actions already drive the Territory Movement grid with net ambience scores per territory, pending indicators, and influence spending totals [ref: downtime_helper/js/dashboard.js, `renderAmbienceScores()`]
- ✅ All submission data (projects, sphere actions, contacts, retainers, acquisitions) is stored in IndexedDB and available via `window._submissions`

What this story adds:
- An approval workflow overlay in the Players tab
- Writing approved outcomes back to `tm_chars_db` in localStorage
- Per-character resolution summary

## Acceptance Criteria

1. Each character's detail panel in the Players tab has an Approval panel with three states per submission: **Pending**, **Approved**, **Modified** (with ST note field), **Rejected** (with reason field).
2. Approval state and ST notes persist in IndexedDB across page reloads.
3. An "Apply All Approved" button (with confirmation dialog) writes approved and modified outcomes to the character's entry in `localStorage` `tm_chars_db`:
   - XP spend requests: appended to an `xp_log.entries` array `[{ date, type: 'downtime', amount, note }]` on the character (see Dev Notes on schema extension).
   - Merit acquisitions: new merit pushed to `char.merits` with correct `category`, `name`, `rating`, and a matching `merit_creation` entry.
   - Territory ambience outcomes: the Territories tab ambience net scores already reflect actual roll results; no further write to character data is required for territory ambience -- it is an in-world state tracked in the downtime helper, not on the character sheet.
4. Rejected outcomes write only an ST note to `submission._raw.st_notes`; character data is not touched.
5. Modified outcomes apply with the ST override note recorded in the XP log entry or merit `notes_col_d` field.
6. After applying, the system generates a per-character resolution summary object listing all changes made (XP entries added, merits added).
7. The resolution summary is displayed in the dashboard after application and can be exported as `downtime_resolution_YYYY-MM-DD.json`.
8. Applying a second time for the same cycle is blocked with an "Already applied" warning; a "Reset applied state" option is available for ST use.
9. The main SPA reflects applied changes when the character is next loaded (character renders with new merits/XP in editor).

## Dev Notes

### Source Tree Locations

- Character data: `localStorage` key `tm_chars_db`, v2 JSON array [ref: data-models.md, localStorage Layout]

### XP Log Schema Extension
The current v2 schema has `xp_log.spent` as category totals (flat ints: `attributes`, `skills`, `merits`, etc.) [ref: data-models.md, XP fields]. There is no `entries` array. This story needs to add one.

**Recommended approach**: add `xp_log.entries = []` alongside the existing totals; don't replace them. Entries accumulate; totals remain for backward compatibility with existing rendering code. Each entry:
```json
{ "date": "2026-04-08", "type": "downtime", "amount": 2, "note": "April cycle -- Contacts improvement" }
```
The editor's XP display can be extended to show entries in Story 2.1 or later; for now, simply appending to the array is sufficient.

### Merit Acquisition
- New merits require a parallel `merit_creation` entry at the same index [ref: HANDOVER_v4.md, Section 2, merit_creation parallel array]
- `merit_creation` shape: `{ cp: 0, free: 0, xp: cost, up: 0 }` where `cost = rating * 1` (merits: 1 XP/dot) [ref: data-models.md, XP Cost Rates]
- After adding: run `ensureMeritSync(c)` logic or equivalent to keep the two arrays aligned

### Territory Outcomes -- No Character Write Needed
FR-2-01d territory influence generation is **already handled** by the ambience score system in `renderAmbienceScores()` -- it aggregates roll successes and influence spending into a net score per territory displayed live. This is an ST reference tool, not a character data write. Territory domain merits (influence merit ratings) are not changed automatically by downtime; the ST applies those manually via Story 2.1 (direct stat editing) if needed.

### Approval State DB Notes
- Add `approval_status` and `st_override_note` fields to the `record` object in `upsertSubmission()` [ref: downtime_helper/js/db.js, `upsertSubmission()`]
- On upsert of existing record: PRESERVE existing approval state -- do not reset it when a CSV is re-uploaded
- The upsert currently checks `_rawHash` for unchanged detection; approval state should not affect that hash

### script load order
Add `<script src="js/apply-outcomes.js"></script>` in `downtime_helper/index.html` after `db.js` and before `main.js`.

### Testing
