# Story 2.8: GitHub API Integration -- Save Edits Back to Repository

**Status:** Draft

## Story

**As a** Storyteller,
**I want** to save character edits directly back to the GitHub repository JSON from within the Editor,
**so that** the canonical `chars_v2.json` file is updated without manual export and file replacement.

## Acceptance Criteria

1. A "Settings" panel (or modal) in the Editor allows the ST to enter a GitHub Personal Access Token (PAT); the token is stored only in `localStorage` and is never committed to the repository or logged to the console.
2. A "Save to GitHub" button appears in the Editor after a PAT is configured; it is only visible/enabled when there are unsaved changes (`markDirty()` has been called).
3. Before writing, the system fetches the current file SHA from `GET /repos/{owner}/{repo}/contents/{path}` and compares it to the SHA stored at last load; if they differ, a conflict warning is shown and the write is blocked until the ST confirms override or reloads.
4. On successful save, the system calls `PUT /repos/{owner}/{repo}/contents/{path}` with: the full updated character array as base64-encoded content, the current file SHA, and a commit message in the format `"Update characters: {summary of changes}"`.
5. The commit message summarises which characters were modified (e.g. "Update characters: Viktor Ashwood (merits), Jade Moreno (xp)").
6. After a successful save, the dirty flag is cleared, the stored SHA is updated, and the ST sees a "Saved to GitHub" confirmation.
7. On API error (network failure, auth failure, rate limit), the system displays the HTTP status and message; the dirty flag is NOT cleared and the character data is NOT lost.
8. The repository owner, repo name, and file path are configurable constants in a `public/js/editor/github.js` config block -- not hardcoded across multiple files.
9. No PAT or sensitive value appears in any JS source file committed to the repository.

## Tasks / Subtasks

- [ ] Task 1: PAT settings UI and storage (AC: 1, 9)
  - [ ] Create a Settings modal (or panel) in the Editor, accessible via a gear icon in the header
  - [ ] Input field: "GitHub Personal Access Token" -- type `password` to mask the value
  - [ ] On save: `localStorage.setItem('tm_github_pat', token)` -- store under a non-obvious key
  - [ ] On load: if `localStorage.getItem('tm_github_pat')` is present, show "PAT configured" status (do not display the token value)
  - [ ] Clear PAT button: removes the key from localStorage
  - [ ] Warn the ST: the PAT needs `contents: write` permission on the repository

- [ ] Task 2: GitHub API module (AC: 3, 4, 5, 8)
  - [ ] Create `public/js/editor/github.js` with:
    - Config block: `GITHUB_OWNER`, `GITHUB_REPO`, `GITHUB_FILE_PATH` as constants (e.g. `'angelusvmorningstar'`, `'TerraMortis'`, `'data/chars_v2.json'`)
    - `fetchFileSHA(pat)` -- `GET /repos/{owner}/{repo}/contents/{path}` with `Authorization: token {pat}`; returns `{ sha, content }` or throws on error
    - `pushFileUpdate(pat, content, sha, commitMessage)` -- `PUT /repos/{owner}/{repo}/contents/{path}` with base64-encoded content + sha + message
    - `buildCommitMessage(changedChars)` -- returns human-readable summary string
  - [ ] Base64 encode: use `btoa(unescape(encodeURIComponent(jsonString)))` for Unicode-safe encoding
  - [ ] Store last-known SHA in `localStorage` key `tm_github_sha` at load time

- [ ] Task 3: Conflict detection (AC: 3)
  - [ ] On "Save to GitHub" click: call `fetchFileSHA(pat)` to get current remote SHA
  - [ ] Compare to stored `tm_github_sha`; if different, show modal: "The remote file has been updated since you last loaded. Save anyway? (this will overwrite remote changes)"
  - [ ] If ST confirms override: proceed with PUT using the new remote SHA
  - [ ] If ST cancels: abort; dirty flag remains set

- [ ] Task 4: Save to GitHub button and flow (AC: 2, 6, 7)
  - [ ] Add "Save to GitHub" button to Editor toolbar; show only when PAT is configured
  - [ ] Disable button when no dirty flag is set (`isDirty === false`)
  - [ ] On click: `charsForSave()` to get clean character array [ref: HANDOVER_v4.md, Section 3], JSON.stringify, base64-encode, call `pushFileUpdate()`
  - [ ] On success: clear dirty flag, update `tm_github_sha` in localStorage, show success toast
  - [ ] On error: display HTTP status + error message in an alert/panel; do NOT clear dirty flag; do NOT discard data

- [ ] Task 5: Commit message generation (AC: 5)
  - [ ] Track which character names were modified since last save (dirty tracking per character)
  - [ ] `buildCommitMessage(changedChars)` returns e.g. `"Update characters: Viktor Ashwood (merits), Jade Moreno (xp_log)"`
  - [ ] If more than 5 characters changed: `"Update characters: N characters modified"`

- [ ] Task 6: Manual testing (AC: 6, 7, 9)
  - [ ] Configure PAT in settings; make a character edit; click "Save to GitHub"
  - [ ] Verify GitHub repository file is updated and commit message is correct
  - [ ] Test with an invalid PAT -- verify error message displays and dirty flag remains
  - [ ] Test conflict scenario: manually edit the file on GitHub between load and save; verify conflict warning fires
  - [ ] Confirm no PAT appears in console logs, network request logs (beyond the auth header), or source files

## Dev Notes

### GitHub REST API Endpoints

```request
GET  https://api.github.com/repos/{owner}/{repo}/contents/{path}
  Headers: Authorization: token {pat}
  Response: { sha: string, content: string (base64), ... }

PUT  https://api.github.com/repos/{owner}/{repo}/contents/{path}
  Headers: Authorization: token {pat}, Content-Type: application/json
  Body: { message: string, content: string (base64), sha: string }
  Response: { commit: { sha: string }, ... }
```

[ref: tech-stack.md, Phase 2 Additions: "GitHub REST API -- Editor saves changes back to repo JSON. Client-side only, no server needed."]

### PAT Security

- The PAT must have at minimum `repo` scope (for private repos) or `public_repo` (for public repos) with `contents: write` [ref: epic-2-operational-enhancement.md, FR-2-08 and Technical Notes: "Sensitive credentials must not enter the repository (see NFR7)"]
- `localStorage` storage is acceptable for this use case: the ST owns the device and the PAT is scoped to this repo [ref: tech-stack.md, Design Rationale: "The ST owns the device; localStorage is sufficient for the single-device-at-game use case"]
- **Never** log the PAT to the console or include it in error messages

### Base64 Encoding (Unicode-safe)

```js
// Unicode-safe base64 encoding for JSON content
const encoded = btoa(unescape(encodeURIComponent(jsonString)));
// Decoding:
const decoded = decodeURIComponent(escape(atob(encodedString)));
```

Required because `btoa()` alone fails on non-ASCII characters (e.g. character names with accents, special symbols).

### Dirty Flag

- `markDirty()` is called by `shEdit()` on any character change [ref: CLAUDE.md, Editor control flow]
- The "Save to GitHub" button should observe the same dirty state used by the existing local `saveChars()` flow

### Deployment Context

- `chars_v2.json` lives in `data/` (never in `public/`) [ref: unified-project-structure.md, Deployment Layout]
- The file path for the PUT/GET is `data/chars_v2.json` in the repository root

### Testing

- Phase 2+ note: "GitHub API integration should be tested against a test repository to avoid writes to production data" [ref: testing-strategy.md, Phase 2+ Testing Considerations]
- Manual: create a test fork or branch; run the full save cycle against it before testing against production
- Verify `chars_v2.json` content in GitHub matches `JSON.stringify(charsForSave(), null, 2)` output (no derived merits, correct encoding)
