## 5. Circular Dependencies

| Cycle | Functions |
|---|---|
| Edit -> Render loop | Most `sh*` edit handlers call `renderSheet`, which calls `shRender*` functions, which emit `onclick` attrs that call `sh*` edit handlers. This is not a runtime circular call (the onclick is deferred), but it means edit and sheet modules must be able to reference each other at load time. |
| `applyDerivedMerits` <-> `meritQualifies` | `applyDerivedMerits` calls `meritQualifies` to check prereqs on MCI grants. `meritQualifies` calls `checkSinglePrereq`. No reverse dependency -- not truly circular. |
| `renderSheet` <-> `editFromSheet` | `editFromSheet` calls `renderSheet`; `renderSheet` is called from `editFromSheet`. Not circular at runtime. |
| Domain partner writes | `shAddDomainPartner` and `shRemoveDomainPartner` write to multiple entries in `chars` (not just `chars[editIdx]`), making them cross-cutting. |

**Key insight:** There are no true runtime circular calls. The apparent cycles are all mediated by the DOM event loop (onclick handlers fire later, not synchronously during render). This means modules can import each other as long as the import is resolved before any onclick fires, which it will be.

## 7. Critical Extraction Risks

### 7.1 The `chars` Array is Everywhere
47+ functions read or write `chars`. This must be a single shared reference (module-level `let` in `state.js` re-exported). If any module creates a copy, updates diverge.

### 7.2 Domain Partner Writes Are Cross-Cutting
`shAddDomainPartner` and `shRemoveDomainPartner` write to **other characters' merit arrays** (not just `chars[editIdx]`). They iterate `chars` to find partners by name, modify their `shared_with` arrays, and even call `addMerit`/`removeMerit` on other characters. These functions must have direct access to the shared `chars` array.

### 7.3 All onclick-Referenced Functions Must Be Global
Because the app uses inline `onclick` attributes in dynamically generated HTML, every function listed in Section 4 must be accessible from `window`. The simplest approach: each module exports its public functions, and `main.js` assigns them to `window`.

### 7.4 `editMode` is Read During Render, Written Only by `editFromSheet`
This is a simple flag but it flows into every `shRender*` function. It should live in `state.js`.

### 7.5 `openExpId` Has Hidden Coupling
`toggleExp` reads/writes it, and `renderSheet` resets it to `null`. Both must share the same reference.

### 7.6 Inner Functions in `shRenderDisciplines` and `shRenderAttributes`
`renderDiscRow` (line 550), `renderDiscEditRow` (line 568), and `attrCat` (line 418) are defined inside their parent functions. They can stay as closures or be promoted to module-level if needed for testing.
