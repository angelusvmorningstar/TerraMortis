# Story 2.1: Character Administration -- Direct Stat Editing

**Status:** Draft

## Story

**As a** Storyteller,
**I want** to directly edit Status, Blood Potency, Humanity, features, touchstones, and banes from the character sheet,
**so that** I can make administrative updates without switching to an external tool or manually editing JSON.

## Acceptance Criteria

1. ST can increment/decrement `status.city`, `status.clan`, and `status.covenant` independently using up/down controls; values are clamped to valid ranges and persist to localStorage.
2. ST can edit `blood_potency` via a numeric input validated to the range 0-10; value persists to localStorage.
3. ST can edit `humanity` via a numeric input validated to the range 0-10; value persists to localStorage.
4. The `features` string field renders in view mode for all 5 characters that have it populated; characters without `features` show nothing in that position.
5. ST can add a touchstone (name, description, humanity association 1-10); it appears on the sheet and persists.
6. ST can edit an existing touchstone's name, description, and humanity level; changes persist.
7. ST can remove a touchstone; it is removed from the sheet and localStorage immediately.
8. ST can add a bane (name, effect); it appears on the sheet and persists.
9. ST can edit an existing bane's name and effect; changes persist.
10. ST can remove a bane; it is removed from the sheet and localStorage immediately.
11. All edits route through `shEdit()` / `markDirty()` and are written by `saveChars()` -- no direct property mutation outside the edit module.
12. No console errors on any test character after these changes.

### Data Models
- `status`: `{ city: int, clan: int, covenant: int }` [ref: data-models.md, Character section]
- `blood_potency`: `int`, `humanity`: `int` -- top-level character fields [ref: data-models.md, Character section]
- `features`: `string | null` -- top-level optional field [ref: HANDOVER_v4.md, v2 Schema section]
- `touchstones`: `[{ humanity: int, name: string, desc: string }]` [ref: HANDOVER_v4.md, v2 Schema section]
- `banes`: `[{ name: string, effect: string }]` [ref: HANDOVER_v4.md, v2 Schema section]
- 5 test characters have `features` populated; none have rendered it yet [ref: data-models.md, Known Data Issues; testing-strategy.md, Required Coverage]

### Editor Control Flow
- Edit mode state is managed by `shEdit(field, value)` → `markDirty()` → `saveChars()` [ref: CLAUDE.md, Editor control flow]
- `saveChars()` writes the full character array to `localStorage` key `tm_chars_db` [ref: data-models.md, localStorage Layout]
- `charsForSave()` deep-copies and strips `derived: true` merits before save -- do not bypass this [ref: HANDOVER_v4.md, Section 3, Derived merit system]

### Derived Stats Dependency
- `blood_potency` changes affect Vitae max and Vitae per turn -- these are computed in `derived.js` [ref: data-models.md, Derived Stats table]
- The sheet must call `derived.js` functions at render time; do not store these values [ref: data-models.md, "Derived Stats (never stored)"]

### Display Conventions
- Dots: `'●'.repeat(n) + '○'.repeat(max - n)` using U+25CF / U+25CB [ref: coding-standards.md, Display Conventions]
- British English: "Humanity", "Blood Potency" -- no Americanisms [ref: coding-standards.md, Language and Locale]

### Testing

- Test fixture `chars_test.json` should include a character with `features` populated [ref: testing-strategy.md, Required Coverage]
