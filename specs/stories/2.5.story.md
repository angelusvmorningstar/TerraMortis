# Story 2.5: Downtime -- Feeding Roll Resolution

**Status:** Draft

**Priority: HIGH -- soft deadline 8 April 2026**

## Story

**As a** Storyteller,
**I want** the downtime helper to automatically construct each character's feeding roll pool from their stored character data and roll it against their submitted feeding approach,
**so that** feeding outcomes are consistent, rules-accurate, and require no manual pool lookup.

## Context: What the Downtime Helper Already Has

The downtime helper (`downtime_helper/`) already provides:
- ✅ Full roll infrastructure: `roll_pool()`, dice pool badges, roll modal, rote support (`roller.js`)
- ✅ Roll persistence: `db.saveRoll(cycleId, charName, source, index, rollFields)` writes into `_raw[source][index]`
- ✅ Post-roll display: grey badge + inline successes when a roll is stored
- ✅ Territory movement: ambience score calculation using roll successes already drives the Territories tab

What this story adds is **the bridge from the downtime form submission to character data in the main SPA** so that the feeding pool can be constructed and rolled automatically.

## Acceptance Criteria

1. For each submission in the active cycle, the downtime helper reads `localStorage` key `tm_chars_db` and matches the submission `character_name` to a character (case-insensitive, trimmed).
2. The character's feeding pool is constructed from their character data: the hunting pool expression is derived from the character's attributes, skills, and relevant merits using `getPool(char, poolExpression)` logic consistent with the main SPA's pool parser (`public/js/shared/pools.js`).
3. The approach modifier from the submission's feeding approach field is applied to the pool size before rolling.
4. A "Roll Feeding" button appears per character in the Players tab detail view; clicking it rolls the constructed pool using the existing roll infrastructure and stores the result under source `'feeding'` with index `0`.
5. Characters whose names do not match any entry in `tm_chars_db` show a "Character not found in SPA data" warning badge; no roll is attempted.
6. Characters whose feeding pool cannot be constructed (expression unparseable or no hunting pool found) show a "Pool unknown -- manual roll" warning; the roll badge still appears with a manually-overridable pool size input.
7. Roll results persist in IndexedDB via `db.saveRoll()` and display as a grey "rolled" badge exactly as project/sphere rolls do.
8. A "Roll All Feeding" button in the Players tab header rolls all un-rolled characters in sequence.

```js

char.attributes.Physical.Stamina.dots + char.attributes.Physical.Stamina.bonus

char.skills.Physical.Survival.dots + char.skills.Physical.Survival.bonus
// Blood Potency
char.blood_potency
```
[ref: data-models.md, Character section; schema_v2_proposal.md]

### DicePool Shape (existing, used by dicePoolDisplay)
```js
{ expression: string, size: number | null }
```
`dicePoolDisplay(pool, rollCtx, existingRoll)` in `dashboard.js` handles both rolled and unrolled states already. [ref: downtime_helper/js/dashboard.js]

### saveRoll Signature
```js
saveRoll(cycleId, characterName, source, index, rollFields)
// source: 'projects' | 'sphere_actions' -- currently writes raw[source][index]
// Feeding: raw.feeding_roll = rollFields.roll  (not array-indexed)
```
[ref: downtime_helper/js/db.js, `saveRoll()`]

### Script Loading Order
The downtime helper loads scripts as plain globals (no ES modules) so it works on `file://` [ref: downtime_helper/README.md].
`char-bridge.js` must load before `dashboard.js`. Add to `downtime_helper/index.html`:
```html
<script src="js/char-bridge.js"></script>  <!-- before dashboard.js -->
```
