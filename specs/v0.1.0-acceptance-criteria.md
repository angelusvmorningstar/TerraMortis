## Purpose

The bar against which every audit finding triages. If a finding satisfies a criterion below and is currently failing, it is a `v0.1.0-blocker`. Otherwise it is `post-v0.1.0`. No third category.

This document and the pre-cycle hotfix list (`specs/pre-cycle-hotfix-list.md`) together define the scope of the freeze's stabilisation pass. Anything outside both is deferred to the post-v0.1.0 stakeholder meeting Symon has flagged.

## v0.1.0 in one line

A useful tool for the 2026-05-07 DT cycle: every player can read their sheet correctly, submit their DT in-app, and trust that what the app computes is what their sheet says.

## 1. Character sheet correctness

### Must hold for every active character in MongoDB `tm_suite.characters`

- **Render path is clean.** Sheet loads without console errors. Sections (attributes, skills, disciplines, merits, devotions, manoeuvres) render their listed entries without missing rows or undefined values.
- **Display name is correct.** `displayName(c)` produces honorific + (moniker || name); admin grid sort uses `sortName(c)`.
- **Dot displays match data.** Inherent dots (●) and derived dots (○) reflect storage; bonus dots are visibly counted; specs render under skills that have them.
- **XP block is internally consistent.** `xpEarned()`, `xpSpent()`, `xpLeft()` all render; the three reconcile to source data; no character shows negative `xpLeft()` without a documented reason.
- **Contacts dots render.** Hotfix item #6.
- **XP misbehaviour fixed for the affected sheet(s).** Hotfix item #7.

## 2. DT form parity with the Google Form

### Must hold for the DT submission form to replace the Google Form

- **A player can complete a full DT submission end-to-end** without losing state, double-saving accidentally, or hitting a flow that locks them out of a valid choice.
- **The DT form covers the same submission domains as the Google Form:** standout moment, feeding actions, project actions, merit-driven actions, support assets (joint vs solo), rumours.
- **Data writes are non-corrupting.** A submitted DT round-trips: the saved document reflects what the player saw on screen, and ST processing reads the same shape.
- **All hotfix items 1–5, 9, 10 resolved:** feeding hysteresis (#1), support assets dismiss (#2), joint-action stickiness (#3), feeding-method discipline lock (#4), rote feed copy errata (#5), action tab scroll (#9), feeding rights tab order (#10).

### Already-deferred items per `pre-cycle-hotfix-list.md` Out-of-scope

Player-selection autocomplete redesign, single-vs-dual roll labels, persistent action rail, tooltip coverage, phone/mobile optimisation. These do not return for v0.1.0.

### NOT in v0.1.0

- The 34-story DT overhaul backlog (DTSR, DTIL, and siblings).
- NPC/edges/relationships DT integration (feature-flagged off per charter).
- New schema fields, new collections, or new helper modules in the DT path (freeze gate item 2).
- DT report v2 structural improvements. The report is downstream of submission; v0.1.0 covers submission ingestion only.

## 3. Rule implementation bar

### Must hold for v0.1.0

- **Dice pool calculations** read effective rating, not inherent only. Bonus dots, specs, `free_*` grants, and standing-merit grants (`_pt_`, `_mci_`, `_ohm_`) all count where the rules say they should.
- **Influence total** uses the canonical formula in `reference_influence_formula.md` (effective rating; all influence merits count, including Resources/Retainer/Staff/Mentor/Attache).
- **Vitae deficit, frenzy resistance, predatory aura, manoeuvre advancement** compute against effective rating, not inherent.
- **Errata is applied** in pool calculations where memory entries (`reference_merits_errata.md`, `feedback_errata_priority.md`) specify.

### The effective-rating tension — UNRESOLVED, FLAGGED FOR END-OF-AUDIT DECISION

The audit (Phase 1) is expected to find that pools and prereqs under-count effective rating in N call sites because no canonical effective-rating helper exists. Three resolution options, to be chosen jointly by Angelus and Peter at end of audit:

1. **Narrow exception.** Permit a single `effective.js` helper as a freeze-permitted hotfix because rule correctness is load-bearing for v0.1.0. Rationale documented inline. Risk: precedent erosion of freeze gate item 2.
2. **In-place fixes only.** Fix the worst N inline call sites without consolidating. Ships under the gate as written. Cost: re-derives the exact drift the audit just identified; partial coverage likely.
3. **Defer to post-v0.1.0.** Ship v0.1.0 with documented under-count bugs in specific pools. Cost: Peter's "every rule implemented" bar slips for those pools.

Decision owner: Angelus, with Peter's advice, at end of Phase 1 audit. This document updates with the chosen option and rationale before the criteria are locked.

### NOT in v0.1.0

- A complete derivations layer covering health/vitae/WP/defence/size/speed/influence as a unified canonical module.
- Test-parity verification between pre- and post-fix derivation outputs (no test framework yet).
- Rule coverage beyond the live sheet, DT, and tracker paths.
