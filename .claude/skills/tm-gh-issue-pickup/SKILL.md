# TM GH Issue Pickup

The "pop the next item off the queue" skill. Takes a GitHub issue, sets up the local environment for work on it, and hands the issue's substance to BMAD SM so a story file is generated with full context.

## When to Use

Invoke when the user asks for any of:
- "Pick up issue N" / "start work on #N" / "let's do issue N" / "begin issue N"
- "What's next?" *if and only if* the user has just looked at the issue list and is choosing one (otherwise just `gh issue list`)
- "Triage and start the next high-priority issue" (skill picks the top one after surfacing the list)

Do **not** invoke for:
- Reading an issue (use `gh issue view N` directly)
- Closing an issue (use `gh issue close` or `tm-gh-pr-for-branch` which closes via PR merge)
- Creating an issue (that's `tm-gh-issue-create`)

## Inputs

- **Issue number** (required). Either explicit (`#14`, `issue 14`) or implied if the user just chose one from a list this skill displayed.
- **Optional flags**:
  - `--branch <name>` → override auto-derived branch name
  - `--base <branch>` → branch off something other than the user's working branch (default: current `Morningstar`/`Piatra`/`piatra`, fallback `dev`)
  - `--no-bmad` → skip the SM hand-off (just create the branch and stop)
  - `--draft` → print plan, don't execute

## Steps

1. **Verify prerequisites:**
   - `gh auth status` succeeds. If not, abort with "run `gh auth login`".
   - Working tree is clean (`git status --porcelain` empty), or only contains files the user is OK to bring along. If dirty, ask before proceeding.
2. **Fetch the issue:**
   - `gh issue view <n> --json number,title,body,labels,assignees,state`
   - Verify state is `open`. If `closed`, ask the user whether they really want to reopen + work on it.
3. **Determine branch base:**
   - If `--base` given, use it.
   - Else: current branch if it's `Morningstar`, `Piatra`, or `piatra`. Otherwise `dev`.
   - Ensure base is up to date: `git fetch origin && git checkout <base> && git merge origin/<base>` (informational only — do not auto-resolve conflicts; surface them).
4. **Derive branch name:**
   - Format: `<owner>/issue-<n>-<slug>` where:
     - `<owner>` = the user's working branch name lowercased (`piatra`, `morningstar`)
     - `<slug>` = first 4-5 keywords from issue title, lowercased, hyphenated, ≤30 chars
   - Example: issue #14 "SSJ: Herd bonus should also count Status dots" on Piatra → `piatra/issue-14-ssj-herd-status-dots`
   - User's `--branch` override wins.
5. **Print the planned actions** before doing anything destructive:
   ```
   PLANNED:
     base branch: <base>
     new branch: <branch-name>
     issue: #<n> "<title>"
     after branch creation: hand off to bmad-create-story with issue body as context

     git fetch origin
     git checkout <base>
     git pull --ff-only
     git checkout -b <branch-name>

   --- ISSUE BODY (will be passed to BMAD SM) ---
   <issue body>
   -----------------------------------------------
   ```
6. **If `--draft`**, stop here.
7. **Otherwise, ask for confirmation** before creating the branch.
8. **On confirm, execute the branch creation.** If `git pull --ff-only` fails (base branch diverged), surface the conflict and abort — do not auto-merge.
9. **If `--no-bmad`**, stop after branch creation and report:
   ```
   On branch <branch-name>. Issue #<n> body printed above for reference. BMAD hand-off skipped.
   ```
10. **Otherwise, hand off to BMAD SM.** This is a soft hand-off: the skill ends with a clear instruction to the *current Claude session* to invoke `bmad-create-story` (or `bmad-agent-sm` if a more conversational story-creation flow is needed). The output should look like:

    ```
    Branch ready: <branch-name>
    Issue context: #<n> "<title>"

    Next: invoking bmad-create-story with the issue context. The story file will be created under specs/stories/ with frontmatter linking back to issue #<n>.

    [Then in the conversation, the assistant proceeds to invoke the bmad-create-story skill, passing the issue body and number as the source-of-truth input.]
    ```

11. The story file generated by BMAD SM **must** include in its frontmatter:
    ```yaml
    issue: <n>
    issue_url: https://github.com/angelusvmorningstar/TerraMortis/issues/<n>
    branch: <branch-name>
    ```
    The skill ends after surfacing this requirement; the actual frontmatter additions happen during BMAD SM's create-story flow. (If `bmad-create-story` doesn't natively accept these fields, the assistant adds them post-hoc by editing the resulting story file.)

## Boundaries

- **Never** create a branch on `main`. Always branch off a working branch or `dev`.
- **Never** force-checkout over uncommitted work. Ask first.
- **Never** auto-resolve merge conflicts when bringing the base branch up to date. Surface them.
- **Never** assign the issue (avoid bouncing notification noise). User's call.
- **Never** comment on the issue announcing the branch (also noise). The PR opened later by `tm-gh-pr-for-branch` provides the cross-reference.
- If the issue has no body (rare — usually means a one-line bug report), proceed but flag it: BMAD SM will need to expand context from the title alone.
- If a branch matching the slug pattern already exists locally or on origin (someone is already working on it), abort and report.

## Example

**User:** `pick up issue 14`

**Skill:**
1. `gh auth status` ✓.
2. Working tree clean ✓.
3. `gh issue view 14` → title "SSJ: Herd bonus should also count Status dots, not just MCI", body has Background/Current/Desired/Acceptance.
4. Current branch `piatra` → use as base.
5. Branch name: `piatra/issue-14-ssj-herd-status-dots`.
6. Prints planned actions + issue body.
7. User confirms.
8. `git fetch && git checkout piatra && git pull --ff-only && git checkout -b piatra/issue-14-ssj-herd-status-dots`.
9. Reports: "Branch ready: piatra/issue-14-ssj-herd-status-dots. Issue context: #14. Next: invoking bmad-create-story."
10. Assistant invokes `bmad-create-story` with the issue body as the substantive input. SM creates `specs/stories/14-ssj-herd-status-dots.story.md` (or whatever naming SM uses), with frontmatter including `issue: 14`.

## Verification

After running:
- `git branch --show-current` returns the new branch name.
- `gh issue view <n>` still shows `OPEN` (we didn't close anything).
- A story file exists under `specs/stories/` referencing the issue.
- The current Claude session has the BMAD SM agent loaded and ready for the dev cycle.
