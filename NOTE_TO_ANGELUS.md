## Changes Made and Why

### 1. Git Hygiene

It should never be committed -- it is specific to your machine and session. It was previously being tracked, which caused conflicts on pull.

Use this folder for sensitive local files -- PDFs, notes, real character data backups -- that you want to keep alongside the project but never push to origin.

### 2. Data Folder

**`data/chars_v2.schema.json` created** -- a formal JSON Schema (Draft 2020-12) for the v2 character format. This gives you:
- Schema validation in VS Code (add `"$schema": "..."` to `chars_v2.json` to get IntelliSense)
- A single source of truth for what a valid character object looks like
- Something SM/Dev agents can reference when writing data-handling stories

Key design decisions in the schema:
- `covenant_standings` uses open keys (constrained to `"Carthian"/"Crone"/"Invictus"/"Lance"/"Ordo"`) -- these are diplomatic standings with OTHER covenants. Your own covenant's mechanical Status lives in `status.covenant`. They are not double-tracking the same thing.
- `PowerPact` has no `stats` or `pool_size` fields -- Invictus oaths are narrative only.
- All merit optional fields (`area`, `qualifier`, `rank_name`, `cult_name`, `role`) are on a single unified Merit type discriminated by `category`.

**`schema_v2_proposal.md` moved** from the project root into `data/` where it belongs alongside the schema and character files.

### 5. BMAD Configuration Redirected

This is the most important operational change. Your BMAD installation was configured during `bmad-init` to write all output to `_bmad-output/`. That meant any document a BMAD agent created would land there rather than in `specs/` where it should live.

**Two config files were updated:**

`_bmad/core/config.yaml`:
```yaml
output_folder: "{project-root}/specs"   # was: _bmad-output
```

`_bmad/bmm/config.yaml`:
```yaml
planning_artifacts: "{project-root}/specs"          # was: _bmad-output/planning-artifacts
implementation_artifacts: "{project-root}/specs/stories"  # was: _bmad-output/implementation-artifacts
project_knowledge: "{project-root}"                 # was: docs (which doesn't exist)
output_folder: "{project-root}/specs"               # was: _bmad-output
```

**What this means in practice:**
- When you run a BMAD PRD agent, it will write to `specs/prd.md` and read existing docs from `specs/`
- When you run the SM agent to create stories, they will land in `specs/stories/`
- BMAD agents will now discover `CLAUDE.md`, `README.md`, `HANDOVER_v4.md`, and `integration_plan.md` as input context (because `project_knowledge` now points at the project root rather than a non-existent `docs/` folder)

## If You Have Your Own Uncommitted or Branched Work

The `piatra` branch has been merged into `main`. If you have local changes or your own branch that predates this merge, here is how to incorporate the upstream changes cleanly.

### Scenario A: You are on `main` with no local changes

```sh
git pull origin main
```

Done.

---

### Scenario B: You are on `main` with uncommitted local changes

Stash your work, pull, then restore:

```sh
git stash
git pull origin main
git stash pop
```

If `stash pop` reports conflicts, open the affected files, resolve them (keep both sets of changes), then:

```sh
git add <conflicted-files>
git stash drop
```

---

### Scenario C: You are on your own branch (e.g. `angelus-work`)

Rebase your branch on top of the updated `main`. This replays your commits cleanly on top of the new base:

```sh
git fetch origin
git rebase origin/main
```

If there are conflicts during rebase, Git will pause at each one. Resolve the conflict in the file, then:

```sh
git add <conflicted-file>
git rebase --continue
```

If it gets messy and you want to start over:

```sh
git rebase --abort
```

Then try a merge instead (less clean history but simpler conflict resolution):

```sh
git merge origin/main
```
